The Styling

I chose a Kanban-style because it's the workflow I personally find most effective. I believe it lets users see a large amount of information at a glance without feeling overwhelming.

I decided to split the page into three parts, the sidebar, the toolbar, and the Kanban component which shows all our tasks.

The Architecture

Overall the project stands on two main models: Task and Project. Both types contain a unique ID for every object. A Project is simply a collection of tasks, each held in a Map. Each project stores three maps, Todo, Progress, and Completed. Rather than a single collection, seperating tasks by status simplified rendering each column and made moving tasks between columns straightforward. Doing it this way did introduce more bookkeeping but it made the rest of the design much simpler.

Since every object was assigned a unique ID, each dynamically created DOM element could be given a corresponding custom data attribute (such as data-task-id or data-project-id). This made it straightforward to locate elements in JavaScript, allowing them to be easily updated, moved, or removed as the application's state changed.

State management is one aread I'd like to revisit. The application currently relies on two variables: stagedProject and editingTask. stagedProject represents the currently active project and determines which tasks are rendered and manipulated throughout the application. This allows us to quickly grab a task as we know where it will be. editingTask simply tells us if we are currently editing a task and makes the appropriate changes such as changing the title, project its tied to, due date, priority, or description. Our rollidex of projects is a map, which contains a projectId : project key : value relationship. Like a Project, this allows us to quickly access items within it.

If I were rebuilding this today, I'd introduce a dedicated state object responsible for coordinating updates between the data model and the UI. This would clear up our AppController file which runs the app and make it more concice. I'll likely come back a refactor this as well as a few other things such as splitting up the AppController into more specific pieces that I won't go into detail on for now.
